# 关卡星级算法 — 设计方案

> 版本：v2  ·  2026-07-16  ·  状态：已落地（1/2星公式调整；2026-07-17 倍率调整：简单 2.8/1.4、标准 2.3/1.5、难 2.0/1.6）

## 1. 概述

通关后按"操作效率"评 1-4 星。星级基于 **剩余步数占可省步数的比例**，跨关卡标准统一；难度通过 **步数预算宽紧** 体现（难关给步数少 → 剩步数难高 → 4星难拿）。

## 2. 难度分档

三档：**简单 / 标准 / 难**。

- 优先读关卡 JSON 的 `difficulty` 字段（`'easy'` / `'normal'` / `'hard'`）。
- 字段缺失时，按关卡 ID 默认：

| 关卡 ID | 默认难度 |
|---|---|
| ≤ 20 | 简单 (easy) |
| ≤ 100 | 标准 (normal) |
| > 100 | 难 (hard) |

## 3. 步数与积分

- `N` = 关卡猪数
- `总步数`（stepBonusThreshold）= `N × 总步数倍率`
- `最少通关步数` minSteps = `N × minSteps倍率`（该关物理下限，玩家再省也只能省到这）
- `可省步数` = `总步数 − minSteps`（玩家"操作效率"的真实浮动区间）
- `积分` = `N + 剩余步数`（每剩余 1 步 = 1 分），其中 `剩余步数 = 总步数 − 实际用步数`

**难度倍率表：**

| 档 | 总步数倍率 | minSteps倍率 | 可省倍率 |
|---|---|---|---|
| 简单 | 2.8 | 1.4 | 1.4 |
| 标准 | 2.3 | 1.5 | 0.8 |
| 难 | 2.0 | 1.6 | 0.4 |

> 可省倍率 = 总步数倍率 − minSteps倍率（即 可省步数 = N×可省倍率）。

## 4. 星级算法

通关后，按 **剩余步数占可省步数的比例** 评星。1/2 星基于 3 星积分做相对折算，3/4 星直接按可省比例：

| 星级 | 公式 | 含义 |
|---|---|---|
| 1星 | s3 × 1/4 | 理论底线（通关积分 = N，通常已经 ≥ 2星） |
| 2星 | s3 × 2/3 | 通关保底（通关即 ≥ 2星；需要剩余步数才能 ≥ 3星） |
| 3星 | s3 = N + 可省 × 0.55 | 操作熟练 |
| 4星 | N + 可省 × 0.80 | 接近最优通关（高手成就） |

**等价的积分阈值**（兼容现有 `_starScores` 积分制架构，无需改判定逻辑）：

```
s3 = N + 可省步数 × 0.55
_starScores = [s3/4, s3×2/3, s3, N + 可省×0.80]   （每步1分，均取整到整数）
```

> **设计意图**：1星和2星的阈值固定低于通关积分 N，使得 **通关即保底 2 星**。3 星和 4 星才需要剩余步数，体现操作差距。

## 5. 算例（N = 40）

各星级积分阈值（`_starScores`，每步1分，取整）：

| 档 | 总步数 | 可省 | 1星 | 2星 | 3星 | 4星 | 通关保底 |
|---|---|---|---|---|---|---|---|
| 简单 | 112 | 56 | 18 | 47 | 71 | 85 | ≥1星(40≥47?否→1星) |
| 标准 | 92 | 32 | 15 | 39 | 58 | 66 | ≥2星(40≥39) |
| 难 | 80 | 16 | 12 | 33 | 49 | 53 | ≥2星(40≥33) |

> 通关积分 = N = 40。标准/难关 2星阈值低于 40 → **通关保底 2 星**；简单关因可省较多（2星阈值 47 > 40）通关即 ≥1星，需剩余步数才能达 2星以上。3星和4星需要额外剩余步数。

**4星对应的用步**（约 1.65-1.7 步/猪）：

| 档 | 简单 | 标准 | 难 |
|---|---|---|---|
| 4星积分 | 85 | 66 | 53 |
| 需剩余步数 | 45 | 26 | 13 |
| 4星用步 | 67 | 66 | 67 |
| 最少用步 | 56 | 60 | 64 |
| 超出 | +11 | +6 | +3 |

## 6. 落地说明

1. **关卡 JSON** 新增可选字段 `difficulty`（`'easy'`/`'normal'`/`'hard'`）；未配按 ID 默认（见 §2）。
2. **`_starScores` 默认公式** 改为按难度档倍率算可省，再 `N + 可省×比例` 生成4个阈值。
3. **现有积分制判定逻辑**（`StarScores.getStarTier(achievedScore, _starScores)`）不变，`achievedScore = escapedCount + scoreBonusRemaining` 不变。
4. 关卡 JSON 仍可显式配 `starScores` 数组覆盖默认（逐关精调用）。
5. `minSteps` 当前按难度档倍率；后续若要更精确，可逐关在 JSON 标注真实最少步数覆盖倍率。

## 7. 调参指引（上线后数据驱动）

上线后观察 4星获得率，按需微调 **4星比例**（其他不变）：

| 现象 | 调整 |
|---|---|
| 难关 4星获得率 < 3%（太硬） | 80% → 70%（难关4星变 57步） |
| 简单关 4星获得率 > 60%（太水） | 80% → 90%（简单关4星变 50步） |

1星/2星（3星×1/4 和 3星×2/3）一般不需动，除非希望调整通关保底星级。

---

**设计原则**：难度靠"步数预算宽紧"诚实体现，不靠暗中压低积分；星级标准跨档一致，玩家建立稳定预期（"剩可省的80%=4星"处处适用）。
