# 场外求助（Help-a-Friend）功能设计方案

> 状态：设计稿 v2（6 项决策已锁定，2026-07-20 评审）
> 基于工程实际代码盘点（2026-07-20），可直接落地。
> 配套架构图见同目录 Ardot 设计文件《场外求助功能设计.ardot》。

---

## 0. 一句话概括

玩家在关卡内点「场外求助」→ 把**当前棋盘快照**（自包含、可复现）压缩后存云端、拿到一个 `helpKey` → 通过微信分享卡片把 `helpKey` 发给好友 → 好友点卡片**走正常流程（主菜单+关卡地图预加载），从主菜单的协助邀请进入协助场景**，在棋盘上操作（以棋盘左上角为参考坐标被录制）→ 点「发送给好友」把录制上传并留下昵称/头像 → 玩家在主菜单左侧「奖杯」按钮的「求助信息」列表看到谁帮了、跑出几头猪，点「回放」自动重演好友的解法。

---

## 1. 现有能力盘点（已存在、可直接复用）

设计前先摸清家底，避免重复造轮子：

| 能力 | 位置 | 复用方式 |
|------|------|----------|
| **棋盘状态快照** | `PlayingEngine._buildCheckpoint()`（`js/game/PlayingEngine.js:2231`） | 复用其字段（`levelName/steps/bonusSteps/version/pigs[]/hintCache...`）作为快照基础 |
| **棋盘相对坐标触摸录制** | `PlayingEngine._recordTouch()`（`:1120`，写入 `{type, bx, by, dt}`，`bx/by` 来自 `this.gp.screenToBoard`） | 协助场景直接复用：**录制天然以棋盘左上角为参考坐标**，与需求一字不差 |
| **触摸回放（board→screen 还原）** | `PlayingEngine._trialStartPlayback()`（`:603`） | 玩家侧「回放」直接复用：`sx = xf.screenCX + (bx - xf.boardCX) * xf.scale` |
| **录制缓冲** | `this._recordEntries = [{type,x,y,dt}]`，`_isRecording` 开关（`:108/:110`） | 协助场景开启 `_isRecording`，结束时取 `_recordEntries` 上传（不写本地 storage） |
| **求助按钮入口（已留位）** | `PlayingEngine.onTouchStart` 内 `_uiHelpBtn` 命中分支（`:1074`），当前 `showToast('功能待上线')` | 把这块「待上线」分支接上场外求助流程即可 |
| **云函数范式** | `cloudfunctions/{getPlayerData,savePlayerData,getOpenId}`（均 `cloud.getWXContext().OPENID` + `db.collection('players')`） | 新增 4 个云函数沿用同一范式；跨用户读写在云函数内（服务端 admin 权限）完成 |
| **云调用封装** | `js/cloud.js`（`callFunction` 含并发限流 + `-501003` 退避重试） | 新增接口直接挂到 `cloud.js` 导出，自动获得限流/重试 |
| **解压库** | `js/libs/pako_inflate.min.js`（仅 inflate） | 播放端解压复用；压缩端需补一个 pako deflate 构建（见 §6.4） |
| **全局状态机** | `databus.gameState`（`'menu'/'playing'/'editor'`）+ `databus.returnState` | 新增 `databus.playMode = 'normal'｜'assist'｜'replay'` 区分同一 `playing` 态下的三种子模式 |

**关键结论**：录制/回放内核已 90% 就绪，本功能主要是「接线 + 云端 + 分享路由 + 三个子模式的 UI 收口」，而非从零写引擎。

## 2. 系统架构总览

```
┌──────────────┐         wx.shareAppMessage(query=helpKey)        ┌──────────────┐
│   玩家端      │ ───────────────────────────────────────────────▶│   好友端      │
│ (playing)    │                                                  │ (assist)    │
│  _uiHelpBtn  │                                                  │ 协助场景     │
└──────┬───────┘                                                  └──────┬───────┘
       │ createHelpRequest(snapshot,requester)                         │ getHelpRequest(helpKey)
       │ submitAssist  /  listMyHelpRequests                           │ submitAssist(recording,assistant)
       ▼                                                               ▼
┌──────────────────────────────────────────────────────────────────────────┐
│                          微信云开发（CloudBase）                              │
│  ┌─────────────────┐   ┌──────────────────────────────────────────────┐   │
│  │ 云函数(服务端)   │   │  云数据库 collections                          │   │
│  │ createHelpRequest│   │  help_requests { helpKey, requester,         │   │
│  │ getHelpRequest   │   │     snapshot, assists[≤3], status, expiresAt }│   │
│  │ submitAssist     │   │  players { _openid, helpKeys:[...], ... }     │   │
│  │ listMyHelpRequests│  │                                              │   │
│  └─────────────────┘   └──────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────────────────┘
       玩家端「求助信息 UI」列表 ◀── listMyHelpRequests ──▶ 点「回放」→ replay 场景
```

- **客户端不直接跨 openid 读库**（微信云 DB 安全规则限制：客户端只能读写自己的 `_openid` 记录）。所有跨用户访问都走云函数，在云端用 admin 权限完成。
- 云函数之间无相互调用，全部由客户端经 `wx.cloud.callFunction` 触发，复用 `js/cloud.js` 的限流/重试。

## 3. 数据模型设计

### 3.1 集合 `help_requests`（新建）

| 字段 | 类型 | 说明 |
|------|------|------|
| `_id` | string | 自动生成（或干脆用 `helpKey` 作主键） |
| `helpKey` | string | **全局唯一、不可猜测**；由服务端生成（`hk_` + 随机字节 base36 + 时间戳）。即分享参数 |
| `requesterOpenId` | string | 发起人 openid（`cloud.getWXContext().OPENID`），用于列表查询与去重 |
| `requester` | object | `{ nickName, avatarUrl }` —— 发起时经用户授权取得 |
| lastAssistedBy | string? | 最近一次协助者 openid（去重用） |
| `levelName` | string | 如 `"0007"`，仅用于列表展示上下文 |
| `snapshot` | string | **压缩后 base64** 的棋盘快照（自包含，见 §4.1） |
| `snapshotMeta` | object | 轻量索引，列表无需解压即可展示：`{ steps, totalPigs, escapedPigs, levelName }` |
| `status` | string | `'open'` / `'full'` / `'expired'` / `'cancelled'` |
| `assists` | array | **最多 3 个**；元素结构见下 |
| `createdAt` | date | `db.serverDate()` |
| `expiresAt` | date | `createdAt + 7天`，过期后不再接受协助、列表标记「已过期」 |

**`assists[]` 元素（每个好友一次协助）：**
```jsonc
{
  "assistantOpenId": "oXXXX",          // 去重 + 展示
  "assistant": { "nickName": "小明", "avatarUrl": "https://..." },
  "recordedAt": "<serverDate>",
  "recording": "BASE64(DEFLATE(entries))",  // 棋盘相对坐标触摸序列
  "result": { "escapedPigs": 7, "totalPigs": 12 }  // 该好友帮跑出几头
}
```

### 3.2 集合 `players`（现有，仅加字段）

在 `savePlayerData` 既有逻辑之外，由 `createHelpRequest` 云函数**服务端**原子地把新 `helpKey` 推入：
```
db.collection('players').where({_openid: OPENID})
  .update({ data: { helpKeys: _.push(helpKey) } })
```
- 复用现有 `players` 文档（不新建集合），避免双写版本冲突。
- `helpKeys: string[]` —— 玩家「自己发出去的求助信息的 key 列表」，符合需求原话。

### 3.3 为什么快照要「自包含」而不是只存 diff

现有 `_buildCheckpoint` 只存「猪的位置/朝向」diff，依赖 `levelName` 再去云端拉基础关。但协助场景要求好友**完全离线可复现**且不受「关卡热更新」影响：
- 若只存 levelName + diff，好友端必须与玩家端同一关版本一致，否则棋盘对不齐 → 录制坐标错位。
- 故快照 = `{ board:{boardWidth,oddCols,rows,boardRate}, pigs:[完整字段], steps, bonusSteps, levelName }`，好友端据快照**原地重建棋盘**，不拉云端关，坐标天然一致。（评审确认：不引入 `engineVersion` 参数，避免额外维护逻辑。）

## 4. 三段流程详细设计

### 4.1 玩家侧发起（Flow A）

入口：`PlayingEngine.onTouchStart` 内 `_uiHelpBtn` 分支（`:1074`），把 `showToast('功能待上线')` 替换为 `this._startExternalHelp()`。

```
_startExternalHelp():
  1. 取/授 玩家昵称头像：若本地/玩家档案无 nickName → wx.createUserInfoButton 拉起授权
     （必须是用户手势触发，满足微信隐私合规；缓存到 players）
  2. 构造快照：
     snap = {
       board: this.gp.board,                 // 棋盘参数（oddCols/rows/boardWidth/boardRate）
       pigs: this.gp.pigs.map(p => ({ id,angle,hintAngle,hintId,length,tail,type,collisionWidth })),
       steps: this.steps,
       bonusSteps: this._bonusSteps,
       levelName: this.levelName
     }
  3. 压缩：pako.deflate(JSON.stringify(snap)) → Uint8Array → base64  → snapB64
  4. 发起新求助：每次点「求助」都新建一条 help_requests（不复用旧 key）
     res = cloud.createHelpRequest({ snapshot: snapB64,
            snapshotMeta:{steps,totalPigs,escapedPigs,levelName},
            requester, levelName })
     helpKey = res.helpKey
     cloud 服务端原子把 helpKey 推入 players.helpKeys（同一关可有多条）
  5. wx.shareAppMessage({ title:"帮我过这关！", imageUrl: SHARE_IMG,   // SHARE_IMG = 固定运营图占位 assets/share/help_share.png（后续美化，不用实时截图）
       query: "hk=" + helpKey })   // 选一个好友发送，卡片携带 hk
  6. toast("已发送给好友，等待协助")
```
- **同一关可多次求助**：每点一次「求助」= 新建一条 `helpKey` 并发给一个好友；`players.helpKeys` 可含同一关的多条记录。
- **继续转发**：在「求助信息」列表内，对已有 `helpKey` 点「继续转发」可再次 `wx.shareAppMessage` 发给更多好友，同一 `helpKey` 的 `assists[]` 最多 3 人。
- **快照是「时间点冻结」**：玩家分享后继续自己玩，不影响已发出快照；好友回放的是那个冻结态的解法。

### 4.2 好友侧协助（Flow B）—— 走正常流程（不跳过主菜单/关卡地图）

**路由入口（新增）**：`GameEngine.start()` 冷启动读 `wx.getLaunchOptionsSync().query`；`wx.onShow` 每次读 `options.query`。若含 `hk` → 仅置 `databus._pendingHelpKey`，**仍然正常进入主菜单、关卡地图照常预加载**（不跳过任何环节、不增加复杂度）。

```
_enterAssistFromHelpKey(hk):   // 好友从主菜单的「协助邀请」入口触发
  1. res = cloud.getHelpRequest(hk)
     若 res.code==NOT_FOUND → toast("求助不存在")→ menu
     若 res.status=='expired' → toast("求助已过期")→ menu
  2. 解压：snap = JSON.parse(INFLATE(base64decode(res.snapshot)))
     （复用现有 pako_inflate）
  3. databus.gameState = 'playing'; databus.playMode = 'assist'
  4. PlayingEngine.loadAssistScene(snap):
     - 据 snap.board/pigs 原地重建棋盘（不走关卡地图/云端拉关）
     - UI 层只挂：棋盘 + 右上角剩余步数(复用 RightStepWidget) + 底部正中「发送给好友」按钮
     - 不挂：顶栏设置、提示/求助道具钮、金币、关卡地图、引导手
  5. 开启录制：this._isRecording = true; this._recordEntries = [];
     （复用 _recordTouch，天然录 board 相对坐标）
  6. 好友自由操作；点「发送给好友」→ _submitAssist()
```

主菜单「协助邀请」入口：当 `databus._pendingHelpKey` 存在时，主菜单（与关卡地图并列）展示一张邀请卡（含发起人昵称 + 关卡名 + 「去帮忙」钮），点击即调 `_enterAssistFromHelpKey(_pendingHelpKey)`。好友正常看完/关掉邀请卡也能继续正常玩自己的进度，互不干扰。

```
_submitAssist():
  1. this._isRecording = false
  2. entries = this._recordEntries
  3. result = { escapedPigs: 已逃出猪数, totalPigs: snap.totalPigs }
  4. 取/授 好友昵称头像（wx.createUserInfoButton，手势触发）
  5. recB64 = base64(DEFLATE(JSON.stringify(entries)))
  6. res = cloud.submitAssist({ recording: recB64, result, assistant })
     服务端校验 assists.length<3 且 assistantOpenId 未重复 → 否则回 code=FULL/DUP
  7. 成功 → toast("协助完成，感谢！") → 回主菜单
     （回菜单须先置 databus._returningToMenu=true 再切 gameState，沿用既有过场防闪小手纪律）
```

### 4.3 玩家侧查看与回放（Flow C）

**列表入口**：主菜单左侧的「奖杯」按钮，打开 `HelpListPanel`（复用 `UIManager` 模态层 + `assets` 现有列表控件风格）。

```
打开 HelpListPanel:
  list = cloud.listMyHelpRequests()   // 服务端按 requesterOpenId 查 help_requests
  每项渲染：
    - 关卡名 + 发起时间 + 剩余步数
    - 状态：open(N/3) / full(3/3) / expired
    - 每个 assist：头像 + 昵称 + "帮跑出 X/Y 头猪"
    - 若 assists.length>0 → 显示「回放」按钮（每个 assist 一个，或默认最新一个）
    - 若 assists.length<3 且 status=='open' → 显示「继续转发」按钮（复用 wx.shareAppMessage 带同一 hk）
```

```
点「回放」→ _enterReplayFromHelpKey(hk, assistIndex):
  1. res = cloud.getHelpRequest(hk) → 解压 snap 重建棋盘
  2. databus.gameState='playing'; databus.playMode='replay'
  3. PlayingEngine.loadReplayScene(snap): 只挂棋盘，无交互；底部常驻「返回」钮
  4. 取 assists[assistIndex].recording → 解压 entries
  5. 复用 _trialStartPlayback 内核：把 entries 喂给回放器（绕开本地 storage，直接内存数组）
     —— board 相对坐标经 _xform 还原为屏幕坐标，因棋盘同重建故精确对齐
     —— 不允许快进/拖动，仅自动播完
  6. 回放结束 → 底部出现「再看一次」钮（重播同一 recording）+ 常驻「返回」钮（退出到列表）
     （不使用「谢谢」）
```

## 5. 关键技术决策

### 5.1 分享与 helpKey 路由
- 发起：`wx.shareAppMessage({ query: "hk=" + helpKey })`（用户手势触发，合规）。
- 接收：`wx.getLaunchOptionsSync().query.hk`（冷启）+ `wx.onShow(opt => opt.query.hk)`（热启/从卡片回前台）。
- `helpKey` 由服务端生成，不可猜测，避免被枚举他人求助。

### 5.2 录制坐标体系（核心正确性）
- 直接复用 `_recordTouch`：`bx,by = this.gp.screenToBoard(x,y)` —— **本就是以棋盘左上角为原点的相对坐标**。
- 回放 `_trialStartPlayback`：`sx = xf.screenCX + (bx - xf.boardCX) * xf.scale`，因好友/玩家用同一快照重建棋盘，`_xform` 完全一致 → 跨设备精确对齐。
- **无需自己实现坐标换算**，这是本方案最大的省力点。

### 5.3 引擎版本钉定（已取消）
- 评审决定：**不引入 `engineVersion` 参数**，避免额外维护一份版本号与不一致提示逻辑。快照自包含重建棋盘即可保证坐标对齐；若未来猪物理逻辑大改影响回放，届时再评估加版本号。

### 5.4 压缩方案
- **客户端压缩**：新增 `js/libs/pako.min.js`（含 deflate）。发起端 `pako.deflate(JSON)`→base64；协助端录制同理。
- **客户端解压**：播放端复用现有 `pako_inflate`。
- 备选（若不想加包）：改在云函数内用 node `zlib` 压缩/解压，客户端只传原始 JSON（单次分享流量可接受）。**默认采用客户端 pako**，更贴合需求原话「序列化后，压缩，发送」。
- 体量评估：单关快照（board+pigs+steps）约 1–4KB，录制约数百条 × 每条 ~30B ≈ 数 KB，压缩后更小，远低于云函数入参/DB 文档上限。

### 5.5 隐私合规（昵称/头像）
- 微信小游戏取头像昵称用 **`wx.createUserInfoButton`**（需用户点击手势触发，返回 `{userInfo:{nickName,avatarUrl}}`）。
- 发起求助时授一次（存 players）；好友协助提交时授一次（存 assist.assistant）。
- 仅存展示必需字段，遵守 PIPL / 微信隐私规范；在隐私保护指引中补充「求助功能收集昵称头像用于展示协助者」说明。

### 5.6 跨用户云端读写
- 全部经云函数，服务端 `wx-server-sdk` 默认 admin 权限可跨 `_openid` 查询 `help_requests`。
- 客户端零直连 DB，天然规避安全规则限制。

## 6. 边界与异常处理

| 场景 | 处理 |
|------|------|
| **协助已满 3 人** | `submitAssist` 服务端 `assists.length>=3` → 回 `code=FULL`；好友端提示「该求助已满 3 人协助」 |
| **同一好友重复协助** | 服务端按 `assistantOpenId` 去重，重复 → `code=DUP` |
| **求助过期（>7天）** | `getHelpRequest` 返回 `status=expired`；好友/回放端提示已过期 |
| **求助被发起人取消** | 新增 `cancelHelpRequest(helpKey)` 云函数，置 `status=cancelled` |
| **网络失败/限流** | 复用 `js/cloud.js` 的 `-501003` 退避重试；分享失败 toast 引导重试 |
| **好友端解压/重建失败** | try-catch → toast「求助数据损坏」→ 回菜单，不影响主流程 |
| **玩家分享后自己继续玩** | 快照是冻结态，互不影响（设计如此） |
| **回放中玩家误触** | replay 场景屏蔽所有棋盘交互，仅底部「返回」常驻可点；「再看一次」仅在播完后出现 |

## 7. 性能与包体积

- **启动**：好友端与正常玩家完全一致——照常进主菜单、关卡地图预加载；仅当好友主动点「协助邀请」后才重建单关棋盘进入协助场景，不额外优化也不额外负担。
- **包体积**：新增 `pako.min.js`（~10–30KB gzipped 内的 deflate 部分）、若干 UI 面板（复用现有控件，增量小）。小游戏主包上限 4MB、总 8MB（含分包），余量充足；上线前用 DevTools 审计。
- **DB 增长**：`help_requests` 带 `expiresAt`，可定时清理（云函数定时触发器或人工）。
- **并发**：`js/cloud.js` 已限流 3 路 + 重试，分享/协助高峰不击穿云配额。

## 8. 已锁定设计决策点（2026-07-20 评审）

以下 6 项原待确认点已拍板，方案据此定稿：

1. **不引入 `engineVersion`**：快照自包含重建即可，不再维护版本号与不一致提示逻辑。
2. **协助者走正常流程**：好友点卡片后**照常进主菜单 + 关卡地图预加载**，不跳过；从主菜单「协助邀请」卡进入协助场景，不增加复杂度。
3. **求助记录入口 = 主菜单左侧「奖杯」按钮**：直接复用该按钮打开 `HelpListPanel`。
4. **分享卡片图 = 固定运营图**：先用一张占位图（`assets/share/help_share.png`，后续美化），不用实时截图。
5. **同一关可多次求助 + 继续转发**：每点一次「求助」= 新建一条 `helpKey` 发给一个好友；列表内可「继续转发」同一 `helpKey` 给更多人，单 `helpKey` 的 `assists[]` **最多 3 人**。
6. **回放仅自动播完，无快进**：播完底部出现「再看一次」；常驻「返回」钮（不用「谢谢」）。

## 9. 落地步骤拆解（实施计划）

1. **云函数**（新建）：`createHelpRequest` / `getHelpRequest` / `submitAssist` / `listMyHelpRequests` /（可选）`cancelHelpRequest`；`players` 集合加 `helpKeys` 字段写入。
2. **压缩库**：加入 `pako.min.js`（deflate）。
3. **`js/cloud.js`**：导出上述 4–5 个调用封装。
4. **快照/录制复用**：抽一个 `RecordingSession` 薄封装，统一 trial / assist / replay 三处的 `_recordEntries` 取用（最小改动，先不动现有 trial 逻辑）。
5. **路由**：`GameEngine` 加 `getLaunchOptionsSync`/`onShow` 的 `hk` 解析（仅置 `_pendingHelpKey`，不跳过主菜单）；主菜单挂「协助邀请」卡触发 `_enterAssistFromHelpKey`；`_enterReplayFromHelpKey` 由奖杯列表「回放」触发。
6. **`PlayingEngine` 子模式**：`playMode` 分支控制 UI 收口（assist/replay 隐藏非必要控件）+ `loadAssistScene`/`loadReplayScene`。
7. **UI**：`HelpListPanel`（奖杯入口）+ 协助场景「发送给好友」钮 + 列表「继续转发」钮 + 回放「再看一次」/「返回」钮（均复用 `UIManager` 模态/控件与主题字体）。
8. **隐私**：`wx.createUserInfoButton` 接入 + 隐私指引文案。
9. **联调**：冷启/热启分享卡片、3 人上限、过期、回放对齐、版本不一致提示。
10. **提审**：确认分享类目合规、隐私声明完整。
