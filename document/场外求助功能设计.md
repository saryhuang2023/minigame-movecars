# 场外求助（Help-a-Friend）功能设计文档

> 状态：**当前实现态**整理版（2026-07-22 收口，全部功能已实现并通过测试）
> 本文档取代 2026-07-20 设计稿 v2，修正了实现过程中已演进的架构与决策（快照归一、压缩上移服务端、提交改覆盖、UI 纪律全量、触控分发拆分等）。
> 配套策划视角文档见同目录《场外求助系统-策划案.md》。

---

## 0. 一句话概括

玩家在关卡内点「场外求助」→ 把**当前棋盘残局快照**（自包含、可离线复现）压缩后存云端、拿到一个 `helpKey` → 通过微信分享卡片把 `helpKey` 发给好友 → 好友点卡片**走正常流程（主菜单+关卡地图预加载），从主菜单的协助邀请进入协助场景**，在残局棋盘上操作（以棋盘数据坐标为基准被录制）→ 点「发给好友」把录制上传并留下昵称/头像 → 玩家在主菜单「奖杯」按钮的「求助信息」列表看到谁帮了、跑出几头猪，点「回放」自动重演好友的解法。

---

## 1. 关键概念：三子模式与 playMode

整个系统共享同一个 `playing` 游戏态（`databus.gameState === 'playing'`），靠 `databus.playMode` 区分三种子模式：

| 模式 | `playMode` | 谁来操作 | 是否录制 | 是否落库 | 关卡数据来源 |
|------|-----------|---------|---------|---------|------------|
| 正常游玩 | `normal` | 玩家自己 | 可选（提示录制） | 是 | 工程关卡文件 |
| 协助好友 | `assist` | 协助者 | 是（必录） | 否（仅回传云端） | 残局快照还原 |
| 回放复盘 | `replay` | 无（自动播放） | 否 | 否 | 残局快照还原 |

**铁律**：`playMode` **仅作 UI/行为判别器**，不参与加载控制流（加载控制流由 `databus.currentLevel` 真值决定，见 §7.3）。协助/回放复用既有的 `_recordTouch()` 录制内核与 `_trialStartPlayback()` 回放内核。

## 2. 现有能力盘点（已存在、直接复用）

| 能力 | 位置 | 复用方式 |
|------|------|----------|
| 棋盘状态快照 | `PlayingEngine._buildCheckpoint()` | 复用其字段（`levelName/steps/bonusSteps/version/pigs[]/hintCache...`）作为快照基础 |
| 棋盘相对坐标触摸录制 | `PlayingEngine._recordTouch()` | 协助场景直接复用：录制以**棋盘数据坐标**为基准（见 §7.2 跨设备铁律） |
| 触摸回放（board→screen 还原） | `PlayingEngine._trialStartPlayback()` | 玩家侧「回放」直接复用 |
| 录制缓冲 | `this._recordEntries = [{type,x,y,dt}]`，`_isRecording` 开关 | 协助场景开启 `_isRecording`，结束时取 `_recordEntries` 上传（不写本地 storage） |
| 求助按钮入口 | `PlayingEngine.onTouchStart` 内 `_uiHelpBtn` 命中分支 | 接上场外求助流程 |
| 云函数范式 | `cloudfunctions/{getPlayerData,savePlayerData,getOpenId}`（`cloud.getWXContext().OPENID` + `db.collection('players')`） | 新增 4 个云函数沿用同一范式；跨用户读写在云端 admin 权限完成 |
| 云调用封装 | `js/cloud.js`（`callFunction` 含并发限流 + `-501003` 退避重试） | 新增接口直接挂到 `cloud.js` 导出 |
| 解压库 | `js/libs/pako_inflate.min.js`（仅 inflate） | 播放端解压复用；**压缩端在服务端用 node `zlib` 完成**（见 §7.5） |
| 全局状态机 | `databus.gameState`（`'menu'/'playing'/'editor'`）+ `databus.returnState` | 新增 `databus.playMode` 区分同一 `playing` 态下的三种子模式 |

**关键结论**：录制/回放内核已 90% 就绪，本功能主要是「接线 + 云端 + 分享路由 + `_installHelpLevel` 归一 + 三个子模式的 UI 收口」。

## 3. 系统架构总览

```
┌──────────────┐         wx.shareAppMessage(query=helpKey)        ┌──────────────┐
│   玩家端      │ ───────────────────────────────────────────────▶│   好友端      │
│ (playing)    │                                                  │ (assist)    │
│  _uiHelpBtn  │                                                  │ 协助场景     │
└──────┬───────┘                                                  └──────┬───────┘
       │ createHelpRequest(snapshot,requester)                         │ getHelpRequest(helpKey)
       │ submitAssist  /  listMyHelpRequests                           │ submitAssist(recording,assistant)
       ▼                                                               ▼
┌──────────────────────────────────────────────────────────────────────────┐
│                          微信云开发（CloudBase）                              │
│  ┌─────────────────┐   ┌──────────────────────────────────────────────┐   │
│  │ 云函数(服务端)   │   │  云数据库 collections                          │   │
│  │ createHelpRequest│   │  help_requests { helpKey, requester,         │   │
│  │ getHelpRequest   │   │     snapshot, assists[≤3], status, expiresAt }│   │
│  │ submitAssist     │   │  players { _openid, helpKeys:[...], ... }     │   │
│  │ removeAssist     │   │  （发起人删协助/删空移除整条；协助者仅删自己那条）│   │
│  │ listMyHelpRequests│  │                                              │   │
│  └─────────────────┘   └──────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────────────────┘
       玩家端「求助信息 UI」列表 ◀── listMyHelpRequests ──▶ 点「回放」→ replay 场景
```

- **客户端不直接跨 openid 读库**（微信云 DB 安全规则限制：客户端只能读写自己的 `_openid` 记录）。所有跨用户访问都走云函数，在云端用 admin 权限完成。
- 云函数之间无相互调用，全部由客户端经 `wx.cloud.callFunction` 触发，复用 `js/cloud.js` 的限流/重试。

## 4. 数据模型设计

### 4.1 集合 `help_requests`（新建）

| 字段 | 类型 | 说明 |
|------|------|------|
| `helpKey` | string | **全局唯一、不可猜测**；由服务端生成（`hk_` + 随机字节 base36 + 时间戳）。即分享参数 |
| `requesterOpenId` | string | 发起人 openid（`cloud.getWXContext().OPENID`），用于列表查询与去重 |
| `requester` | object | `{ nickName, avatarUrl }` —— 发起时经用户授权取得 |
| `levelName` | string | 如 `"0007"`，仅用于列表展示上下文 |
| `snapshot` | string | **服务端 `zlib.deflateSync` 压缩后**的棋盘快照（自包含，见 §5.1） |
| `snapshotMeta` | object | 轻量索引，列表无需解压即可展示：`{ steps, totalPigs, escapedPigs, levelName }` |
| `status` | string | `'open'` / `'full'` / `'expired'` / `'cancelled'` |
| `assists` | array | **最多 3 个**；元素结构见下 |
| `createdAt` | date | `db.serverDate()` |
| `expiresAt` | date | `createdAt + 7天`，过期后不再接受协助、列表标记「已过期」 |

**`assists[]` 元素（每个好友一次协助）：**
```jsonc
{
  "assistantOpenId": "oXXXX",          // 去重 + 展示
  "assistant": { "nickName": "小明", "avatarUrl": "https://..." },
  "recordedAt": "<serverDate>",
  "recording": "BASE64(DEFLATE(entries))",       // 棋盘数据坐标触摸序列（服务端压缩）
  "result": { "escapedPigs": 7, "totalPigs": 12 },  // 该好友帮跑出几头
  "overwritten": false                            // 覆盖旧录制时由服务端置 true
}
```

### 4.2 集合 `players`（现有，仅加字段）

由 `createHelpRequest` 云函数**服务端**原子地把新 `helpKey` 推入：
```
db.collection('players').where({_openid: OPENID})
  .update({ data: { helpKeys: _.push(helpKey) } })
```
- 复用现有 `players` 文档（不新建集合），避免双写版本冲突。
- `helpKeys: string[]` —— 玩家「自己发出去的求助信息的 key 列表」。

### 4.3 为什么快照要「自包含」而不是只存 diff

现有 `_buildCheckpoint` 只存「猪的位置/朝向」diff，依赖 `levelName` 再去云端拉基础关。但协助场景要求好友**完全离线可复现**且不受「关卡热更新」影响：
- 若只存 levelName + diff，好友端必须与玩家端同一关版本一致，否则棋盘对不齐 → 录制坐标错位。
- 故快照 = `{ board:{boardWidth,oddCols,rows,boardRate}, pigs:[完整字段], steps, bonusSteps, levelName, escapedCount, totalPigs }`，好友端据快照**原地重建棋盘**，不拉云端关，坐标天然一致。（评审确认：不引入 `engineVersion` 参数，避免额外维护逻辑。）

## 5. 三段流程详细设计

### 5.1 玩家侧发起（Flow A）

入口：`PlayingEngine.onTouchStart` 内 `_uiHelpBtn` 分支（每关 2 次配额）。

```
_startExternalHelp():
  1. 取/授 玩家昵称头像：若本地/玩家档案无 nickName → wx.createUserInfoButton 拉起授权
     （必须是用户手势触发，满足微信隐私合规；缓存到 players）
  2. 构造快照：
     snap = {
       board: this.gp.board,                 // 棋盘参数（oddCols/rows/boardWidth/boardRate）
       pigs: this.gp.pigs.map(p => ({ id,angle,hintAngle,hintId,length,tail,type,collisionWidth })),
       steps: this.steps,
       bonusSteps: this._bonusSteps,
       levelName: this.levelName,
       escapedCount: this._escapedCount,     // 残局已逃出猪数
       totalPigs: this._totalPigs            // 残局总猪数（命名按实现）
     }
  3. 上行：客户端发明文 JSON 直接发给云函数（**压缩在服务端完成**，见 §7.5）
  4. 发起新求助：每次点「求助」都新建一条 help_requests（不复用旧 key）
     res = cloud.createHelpRequest({ snapshot: snapJSON,
            snapshotMeta:{steps,totalPigs,escapedPigs,levelName},
            requester, levelName })
     helpKey = res.helpKey
     cloud 服务端原子把 helpKey 推入 players.helpKeys（同一关可有多条）
  5. wx.shareAppMessage({ title:"帮我过这关！", imageUrl: SHARE_IMG,   // SHARE_IMG = 固定运营图占位 assets/share/help_share.png
       query: "hk=" + helpKey })   // 选一个好友发送，卡片携带 hk
  6. toast("已发送给好友，等待协助")
```

- **同一关可多次求助**：每点一次「求助」= 新建一条 `helpKey` 并发给一个好友；`players.helpKeys` 可含同一关的多条记录。
- **继续转发**：在「求助信息」列表内，对已有 `helpKey` 点「继续转发」可再次 `wx.shareAppMessage` 发给更多好友，同一 `helpKey` 的 `assists[]` 最多 3 人。
- **快照是「时间点冻结」**：玩家分享后继续自己玩，不影响已发出快照；好友回放的是那个冻结态的解法。

### 5.2 好友侧协助（Flow B）—— 走正常流程（不跳过主菜单/关卡地图）

**路由入口（新增）**：`GameEngine.start()` 冷启动读 `wx.getLaunchOptionsSync().query`；`wx.onShow` 每次读 `options.query`。若含 `hk` → 仅置 `databus._pendingHelpKey`，**仍然正常进入主菜单、关卡地图照常预加载**（不跳过任何环节、不增加复杂度）。

```
_enterAssistFromHelpKey(hk):   // 好友从主菜单「协助邀请」入口触发
  1. res = cloud.getHelpRequest(hk)
     —— 进入分支判定见 §5.4（已协助/已满/过期/异常）
  2. 解压：snap = JSON.parse(INFLATE(base64decode(res.snapshot)))（复用 compress.inflateJson）
  3. databus.playMode = 'assist'
  4. _proceedAssistLoad：inflate→_buildHelpLevelData→_installHelpLevel(levelData,'assist')
     - 据 snap.board/pigs 原地重建棋盘（不走关卡地图/云端拉关）
     - 存 databus._helpRequester（昵称/头像）+ 复位 _helpRequesterImg
     - UI 层只挂：棋盘 + 右上角剩余步数(复用 RightStepWidget) + 底部正中「就帮到这吧」按钮
     - 不挂：顶栏设置、提示/求助道具钮、金币、关卡地图、引导手
     - gameState==='playing' ? startLevel : 置 gameState='playing'（again 重进修复）
  5. 开启录制：this._isRecording = true; this._recordEntries = [];
     （复用 _recordTouch，天然录棋盘数据坐标）
  6. 好友自由操作；结束 → 点结束面板「发给好友」→ _submitAssist()
```

主菜单「协助邀请」入口：当 `databus._pendingHelpKey` 存在时，主菜单（与关卡地图并列）展示一张邀请卡（含发起人昵称 + 关卡名 + 「去帮忙」钮），点击即调 `_enterAssistFromHelpKey(_pendingHelpKey)`。好友正常看完/关掉邀请卡也能继续正常玩自己的进度，互不干扰。

```
_submitAssist():
  1. this._isRecording = false
  2. entries = this._recordEntries
  3. result = { escapedPigs: 已逃出猪数, totalPigs: snap.totalPigs }
  4. 取/授 好友昵称头像（wx.createUserInfoButton，手势触发）
  5. recB64 = base64(entries JSON)   // 压缩在服务端完成
  6. res = cloud.submitAssist({ recording: recB64, result, assistant })
     服务端：已协助则覆盖（findIndex→filter+push，返 overwritten:true，不占额外名额）；
             满员检查降为安全网（新协助者返 code 3）
  7. 点「发给好友」成功后面板 morph（见 §8.2）：文案变「该帮的也帮了，该我玩了」
     + 中钮变「去玩玩」+ 两侧图标隐藏 → 点「去玩玩」回主菜单
     （回菜单须先置 databus._returningToMenu=true 再切 gameState，沿用黑屏纪律 §7.4）
```

### 5.3 玩家侧查看与回放（Flow C）

**列表入口**：主菜单左侧的「奖杯」按钮，打开 `HelpListPanel`（复用 `UIManager` 模态层 + 现有列表控件风格，双页签「我发出的 / 我协助的」）。

```
打开 HelpListPanel:
  list = cloud.listMyHelpRequests()   // 服务端按 requesterOpenId 查 help_requests
  每项渲染：
    - 关卡名 + 发起时间 + 剩余步数
    - 状态：open(N/3) / full(3/3) / expired
    - 每个 assist：头像 + 昵称 + "帮跑出 X/Y 头猪"
    - 若 assists.length>0 → 显示「回放」按钮（每个 assist 一个，或默认最新一个）
    - 若 assists.length<3 且 status=='open' → 显示「继续转发」按钮
```

```
点「回放」→ _enterReplayFromHelpKey(hk, assistIndex):
  1. res = cloud.getHelpRequest(hk) → 解压 snap 重建棋盘
  2. databus.playMode='replay'
  3. _installHelpLevel(levelData,'replay')：只挂棋盘，无交互；底部常驻「返回」钮
  4. 取 assists[assistIndex].recording → 解压 entries
  5. 进关置 _replayCounting=true + _replayCountdownEnd=now+5000（正中 5 秒倒计时圈）
     倒计时结束 → _trialStartPlayback（自动播完，不允许快进/拖动）
  6. 回放结束 → 底部出现「再看一次」钮（走完整过场重播）+ 常驻「返回」钮
```

### 5.4 进入分支处理（协助入口）

好友点卡片进协助关时，`getHelpRequest` 返回 `alreadyAssisted` / `isFull` 标志位（code 体系 0/1/2/4/-1 不变），客户端据此弹对应分支：

| 状态 | 弹窗类型 | 行为 |
|------|---------|------|
| 正常可协助 | 无弹窗 | 直接进残局开始操作 |
| 已协助过（本人） | `wx.showModal` 确认/取消 | 「这局您已经协助过了，重新协助将覆盖掉旧的协助记录」；确认→`_proceedAssistLoad` 重录，取消→`_exit` |
| 协助者已满（非本人） | 单钮「好的」 | 「协助者已超上限 3 人」→ `_showHelpBlocked` → `_exit` |
| 请求已过期 | 单钮「好的」 | 「请求已过期」→ `_exit` |
| 请求已删除 | 单钮「好的」 | 「该协助已删除」→ `_exit` |
| 数据异常 | 单钮「好的」 | 「该协助异常（{code}）」→ `_exit` |

- **覆盖优先于拒绝**：`submitAssist` 服务端对已协助者执行**覆盖写**（`findIndex→filter+push`，返 `overwritten:true`），不新增名额；满员仅作安全网（新协助者返 `code 3`）。
- 回放读取路径**不判** `alreadyAssisted`/`isFull`，只读 `recording`，不受上述分支影响。

### 5.5 删除协助记录（Flow D）

**入口**：`HelpListPanel` 双页签列表的每一张卡片，在「回放」钮下方新增一个「删除」钮（红描边红字，60×26）。点「删除」先弹**统一确认窗**（见 §7.10），确认后才真正调用云函数删除。

**删除权限矩阵（服务端 `removeAssist` 云函数严格判定，客户端无法越权）：**

| 当前页签 | 删除对象 | `targetOpenId` 传参 | 服务端行为 |
|---------|---------|--------------------|-----------|
| 我发出的（sent） | 该好友的协助条目 | `it.friend.openId`（那条 assist 的 assistantOpenId） | 发起人（`requesterOpenId === OPENID`）可删任意协助者；**若删后 `assists` 为空 → 整条文档 `doc.remove()`**（发起人创建的，有权销毁） |
| 我发出的（sent，无协助者占位） | 整条 noAssist 记录 | `_myOpenId`（兜底） | 同上，直接 `doc.remove()` 整条 |
| 我协助的（assisted） | **仅我自己那条** | `_myOpenId` | 协助者（`assists[].assistantOpenId === OPENID` 且非发起人）**只能删自己那条**；**即使删后 `assists` 为空，也绝不 `doc.remove()` 整条文档**（那是别人发起的，我没权限删别人的东西） |

**铁律（用户明确拍板，务必遵守）**：
- 「我发出的」= 发起人，删空可整条移除，归自己。
- 「我协助的」= 协助者，只能删自己那条；**删完若 `assists` 空，也绝不移除整条文档**（别人发起的记录，无权删）。

**`removeAssist` 云函数返回码：**

| code | 含义 |
|------|------|
| `0` | 成功；`removedDoc:true` 表示整条文档已移除（发起人删空） |
| `1` | 参数缺失 / 无 openid |
| `2` | 求助记录不存在 |
| `3` | `targetOpenId` 不在 `assists` 中 |
| `4` | 不是你可以删的协助（非发起人删他人 / 非本人协助条目） |
| `5` | 协助者试图删他人条目（`targetOpenId !== OPENID`）—— 直接拒绝 |
| `-1` | 服务端异常 |

**客户端调用链（`HelpListPanel`）：**
```
点「删除」→ _confirmDelete(b):
  1. 构造 title='删除这条记录？' + 按 tab 区分的 content
     - sent：带好友昵称 + "删除后该好友的协助信息将移除；若所有好友协助都删光，整条记录会一并删除"
     - assisted：强调"只删除你在这条记录里的协助，发起人创建的记录不会被移除"
  2. ConfirmDialog.open({
       title, content, confirmText:'删除', cancelText:'取消',
       confirmColor:'red', maskClosable:false,
       onConfirm: () => cloud.removeAssist(b.hk, b.targetOpenId)
                        .then(() => open())   // 成功后刷新列表
                        .catch(() => open())  // 失败也刷新（云函数已做权限兜底）
     })
  3. 删除成功后重新拉 listMyHelpRequests 刷新面板
```

## 6. 关键架构决策与实现要点

### 6.1 分享与 helpKey 路由
- 发起：`wx.shareAppMessage({ query: "hk=" + helpKey })`（用户手势触发，合规）。
- 接收：`wx.getLaunchOptionsSync().query.hk`（冷启）+ `wx.onShow(opt => opt.query.hk)`（热启/从卡片回前台）。
- `helpKey` 由服务端生成，不可猜测，避免被枚举他人求助。

### 6.2 录制坐标跨设备铁律（#824）
- `_recordTouch` 存的是**设备无关棋盘数据坐标** `(h.x, h.y)`：
  - 先 `screenToBoard` 取「带偏移渲染屏幕坐标」；
  - 再减 `boardOffsetX` / `(topBarH + boardOffsetY)` 剥离偏移。
- **根因**：`recenterBoard` 的 `boardOffsetX`、`boardOffsetY`、`topBarH(含 safeTop)` 全按**真实屏宽/刘海**算。旧代码把带偏移的屏幕坐标直接存进 `bx/by`，好友用不同屏宽手机回放时差 `(offsetA - offsetB)` 即错位。
- `_trialStartPlayback` 还原须按**当前机** `gp._getAutoScale()` 完整变换（与 `renderBoard` 公式严格一致）：
  - `autoScale < 1`：`SCREEN_W/2 + as*(bx - boardW/2)` / `topBarH + availH/2 + as*(by - visualH/2)`
  - `autoScale >= 1`：`boardOffsetX + bx` / `topBarH + boardOffsetY + by`
- 注意：旧版本地 `trial_record_<level>` 是旧格式，需本机重录一次再生效。

### 6.3 快照自包含重建
- 不引入 `engineVersion` 参数（评审取消），靠快照自包含重建棋盘保证坐标对齐；若未来猪物理逻辑大改影响回放，届时再评估加版本号。

### 6.4 架构归一：试玩通道（根治竞态）
- **根因**：协助/回放曾自发明并行加载机制（`_helpLevelData`）+ `playMode` 过载当控制流标志 → 竞态（空名/静默回菜单/误清 playMode 等坑）。
- **根治**：归一化到试玩通道。
  - `_installHelpLevel(levelData, mode)` 塞 `databus.currentLevel = { name, data }` + `currentLevelIndex = -1` + `playMode = mode`；
  - `startLevel/prepareLevel` 内存关卡判定改 `currentLevel.data` 真值；
  - `activate()` 删 `_helpLevelData` 特判，统一 `_levelReady ? 绑输入 : startLevel(name)`；
  - 回放播放移到 `_afterEnterLevel` 顶部。
- `GameEngine.checkStateTransition` menu 分支 `_resetHelpState` 保留 `prev` 守卫（保住 `_pendingHelpKey`）。`cloud.downloadLevel` 空参护栏保留。

### 6.5 触控分发拆分（#825，函数单一化）
- `handleEvent` 顶层按 `playMode !== 'normal'` 路由：
  - 协助/回放 → `_handleEventAssist`（面板 + `onTouchStart/Move/End`）；
  - 正常模式 → `_handleEventNormal`（VictoryPopup/失败/道具/提示/棋盘，**纯血无 `playMode` 引用**）。
- 两个函数互不感知对方存在；以后改协助 UI 只碰 `_handleEventAssist`，改正常模式只碰 `_handleEventNormal`。

### 6.6 黑屏纪律（#825）
- `PlayingEngine._exit()` 末尾 `databus._returningToMenu` **禁止无条件置 true**。
- `GameEngine.render` 空窗守卫规定 `_returningToMenu=true` 时跳菜单改画 `playing.render()`（冻结帧），该旗仅由「圆形过场结束、menu 激活」（checkStateTransition menu 分支）清回 false。
- 故仅在「当前确为游玩态（`gameState==='playing'`，关卡已加载）返回」时才置 true；**未进关就退出**（协助入口确认弹窗点「取消」/云加载失败/阻塞单钮/入口失败 catch，gameState 本就是 `'menu'` 且没变）必须置 false——否则 `_returningToMenu` 永不清→渲染卡冻结帧且 playing 无关卡可画→黑屏。同一根因也波及 `_showHelpBlocked` 单钮「好的」与所有入口失败 catch，一并修好。

### 6.7 压缩方案（修正：服务端压缩）
- **客户端上行**：发明文 JSON 直接发给云函数（单次分享流量可接受，无需客户端带 pako）。
- **服务端压缩**：云函数内用 node `zlib.deflateSync` 压缩后存 `snapshot` / `assists[].recording`。
- **客户端下行**：播放端用 `compress.inflateJson` 解压（复用现有 pako_inflate 能力，已做小游戏运行时兜底）。
- 体量评估：单关快照（board+pigs+steps）约 1–4KB，录制约数百条 × 每条 ~30B ≈ 数 KB，压缩后更小，远低于云函数入参/DB 文档上限。

### 6.8 隐私合规（昵称/头像）
- 微信小游戏取头像昵称用 **`wx.createUserInfoButton`**（需用户点击手势触发，返回 `{userInfo:{nickName,avatarUrl}}`）。
- 发起求助时授一次（存 players）；好友协助提交时授一次（存 assist.assistant）。
- 仅存展示必需字段，遵守 PIPL / 微信隐私规范；在隐私保护指引中补充「求助功能收集昵称头像用于展示协助者」说明。

### 6.9 跨用户云端读写
- 全部经云函数，服务端 `wx-server-sdk` 默认 admin 权限可跨 `_openid` 查询 `help_requests`。
- 客户端零直连 DB，天然规避安全规则限制。

## 8. 边界与异常处理

| 场景 | 处理 |
|------|------|
| **协助已满 3 人（新协助者）** | `submitAssist` 满员安全网 → 返 `code=3`；好友端提示「该求助已满 3 人协助」 |
| **同一好友重复协助** | 服务端**覆盖**旧录制（`findIndex→filter+push`，返 `overwritten:true`），不占额外名额；进入时弹 `wx.showModal` 确认覆盖/取消 |
| **求助过期（>7天）** | `getHelpRequest` 返回 `status=expired`；好友/回放端提示已过期 |
| **求助被发起人取消（设计预留，未实现）** | 原设计有 `cancelHelpRequest` 云函数置 `status=cancelled`；**当前版本未实现该云函数**，列表过期即自然不展示 |
| **网络失败/限流** | 复用 `js/cloud.js` 的 `-501003` 退避重试；分享失败 toast 引导重试 |
| **好友端解压/重建失败** | try-catch → toast「求助数据损坏」→ 回菜单，不影响主流程 |
| **玩家分享后自己继续玩** | 快照是冻结态，互不影响（设计如此） |
| **回放中玩家误触** | replay 场景屏蔽所有棋盘交互（仅回放自身合成触控放行），仅底部「返回」常驻可点；「再看一次」仅在播完后出现 |
| **协助通关结算面板按钮点不动** | 根因：`handleEvent` 的 `_victory` 守卫曾无条件拦截；已通过 `_handleEventAssist` 独立分支彻底解决（见 §6.5） |
| **我发出的记录：删某好友协助** | `removeAssist(helpKey, friendOpenId)` → 服务端校验发起人权权限删该条目；删后 `assists` 为空则 `doc.remove()` 整条（归发起人） |
| **我协助的记录：删自己那条** | `removeAssist(helpKey, _myOpenId)` → 服务端校验 `targetOpenId===OPENID`；删后**即使 `assists` 为空也绝不 `doc.remove()` 整条**（别人发起的记录，协助者无权删） |
| **协助者试图删他人条目** | 服务端直接拒绝 `code=5`（`targetOpenId !== OPENID`），客户端确认窗不出现该入口 |
| **删除前未二次确认** | 一律先弹统一确认窗（`ConfirmDialog`，§7.10）；`maskClosable:false`，确认才真正调云函数 |

- **并发**：`js/cloud.js` 已限流 3 路 + 重试，分享/协助高峰不击穿云配额。

## 10. 已锁定设计决策点

1. **不引入 `engineVersion`**：快照自包含重建即可，不再维护版本号与不一致提示逻辑。
2. **协助者走正常流程**：好友点卡片后**照常进主菜单 + 关卡地图预加载**，不跳过；从主菜单「协助邀请」卡进入协助场景。
3. **求助记录入口 = 主菜单左侧「奖杯」按钮**：直接复用该按钮打开 `HelpListPanel`（双页签「我发出的 / 我协助的」）。
4. **分享卡片图 = 固定运营图**：先用一张占位图（`assets/share/help_share.png`，后续美化），不用实时截图。
5. **同一关可多次求助 + 继续转发**：每点一次「求助」= 新建一条 `helpKey` 发给一个好友；列表内可「继续转发」同一 `helpKey` 给更多人，单 `helpKey` 的 `assists[]` **最多 3 人**。
6. **回放仅自动播完，无快进**：播完底部出现「再看一次」（走完整过场）；常驻「返回」钮。
7. **重复协助 = 覆盖而非拒绝**：提升协助者体验，名额不浪费（新增决策，取代原 v2 的 DUP 拒绝方案）。
8. **压缩上移服务端**：客户端发明文 JSON，服务端 `zlib.deflateSync`，客户端仅负责解压（取代原 v2 的客户端 pako 方案）。
